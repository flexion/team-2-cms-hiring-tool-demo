# External Boundaries Guidelines

This document explains how to fill in `external-boundaries.md`. The `/program` process reads that file to understand what external systems the application interacts with, so it can create the right interfaces and plumbing modules from the start. The `/mock-context` skill reads it to generate a local mock server that stands in for the real external systems.

## Why This File Matters

The grow-split-merge process enforces the Invasive Species Rule: every external dependency must be isolated behind a wrapper module. It also enforces the Mode of Interaction facet: every library-coupled infrastructure concern needs development plumbing (in-memory test double, used by tests), production plumbing (real library calls, used in production), and — for every external boundary listed in this file — demonstration plumbing (thin HTTP client against the mock server generated by `/mock-context`, used for offline demos).

Without this file, the process discovers external boundaries reactively — a scenario says "sends an email" and eventually a condition checker flags it. By that point, code has been written against the boundary without knowing its shape, and the refactoring to isolate it is more expensive than getting it right from the start.

With this file, the process knows upfront what boundaries exist. When a scenario references an external system, the process creates the domain interface, the development plumbing, the production plumbing, and the demonstration plumbing in the right modules from the first grow step.

## How to Fill In Each Section

### System Name

Name the external system after what it does for your application, not what product it is. "Notification Service" not "SendGrid." "Identity Provider" not "Auth0." The product name goes in the Production Technology field. This keeps the boundary definition stable even if you switch providers.

### Purpose

One sentence: what role does this system play for your application? This orients the developer and helps the process name the interface after a business concept (Screaming Architecture).

### Operations

List every operation your application needs from this system, in domain terms. These become methods on the interface the process creates.

**Data In** and **Data Out** should use your domain language, not the external system's API vocabulary. "Employee name and email" not "POST body with `to` and `subject` fields." The translation between domain terms and API terms is the wrapper module's job — the interface speaks your domain's language.

If you don't know the exact data shape yet, describe it approximately. "Some kind of employee identifier" is better than leaving it blank. The interface will be refined as scenarios drive implementation.

### Testing Considerations

List anything that affects how the development plumbing (the in-memory fake used in tests) should behave. Common considerations:

- **Side effects** — Does the real system do something irreversible? (Sends emails, charges money, publishes to a queue.) The development plumbing must capture the intent without performing the action.
- **Authentication** — Does the real system require credentials? The development plumbing should accept any credentials or none.
- **Failure modes** — Can the real system return errors your application must handle? (Rate limiting, invalid input, network timeouts.) List the failure modes the development plumbing should be able to simulate.
- **Latency** — Is the real system slow enough to affect test speed? The development plumbing should be instant.
- **State** — Does the real system maintain state that your application depends on? (User records, account balances.) The development plumbing needs to simulate that state.

### Status

This tells the process how concrete the boundary is:

- **EXISTS** — There is a real system with an API. The production plumbing will call that API. You may already have API documentation, SDKs, or credentials.
- **PLANNED** — The system will exist but doesn't yet. The process builds the interface and development plumbing now. The production plumbing is deferred until the API is available.
- **ASSUMED** — The scenarios reference this capability, but you haven't decided whether it's a third-party service, a custom-built system, or something else. The process builds the interface and development plumbing. The production plumbing is deferred until you decide. Demonstration plumbing is still buildable for ASSUMED boundaries — either `/program` or `/mock-context` derives the Mock Endpoint Specification from your Operations table and Testing Considerations, and the mock server itself is always local and real. This is a feature of the three-mode model: you can demo the application against a mock for a boundary before you've finalized what the real system is going to be.

### Production Technology

Name the specific product, service, or API if you've chosen one. If you haven't, write "TBD" and optionally reference `tech-stack-recommendations.md` for the category. The process will stop and ask before choosing a technology that isn't listed. The Library field (below) captures the exact package name imported in code — it serves a different purpose than this field.

### Library

Name the exact package(s) or module(s) that code imports to interact with this external system, comma-separated if multiple. For example, if the Production technology is 'SendGrid API', the Library is `@sendgrid/mail`. If the Production technology is 'Stripe', the Library is `stripe`. If a technology has both a client package and a helper package, list both.

This field serves the same purpose as the Library column in tech-stack-recommendations.md: the condition checker uses it to determine which operations in an infrastructure module are directly coupled to an external system's library. Those operations are on the mode-of-interaction axis — for external-boundary libraries they need development, production, and demonstration counterparts (three plumbing modules); for tech-stack libraries they need development and production counterparts (two plumbing modules). Operations that map domain data into the external system's data shapes without importing this library are translation — they don't need separate counterparts.

If the Production technology is TBD, the Library is also TBD. The condition checker cannot evaluate library coupling for boundaries without a known library.

## Mock Endpoint Specification subsection (derived artifact)

The `### Mock Endpoint Specification` subsection under each boundary is a **derived artifact** — its content is computed from your Operations table and Testing Considerations by deterministic rules. You do not write it; either `/program` or `/mock-context` writes it on demand, whichever skill first needs it at that boundary. Both skills apply the same derivation rules (see `${CLAUDE_PLUGIN_ROOT}/references/mock-endpoint-spec-format.md`), so the output is identical regardless of order.

What it documents: the mock server's URL, HTTP methods, request/response shapes, and failure responses for that boundary. `/program` reads it to generate demonstration plumbing (a thin HTTP client that translates domain interface calls into HTTP requests against the mock server). `/mock-context` reads it to generate route handlers that implement those endpoints.

Mechanics:
- The subsection is bounded by `<!-- @mock-context-generated-begin system=<slug> hash=<sha> -->` / `<!-- @mock-context-generated-end system=<slug> -->` markers. Do not hand-edit between the markers — both skills detect hand-edits by hash mismatch and preserve them with a warning, which also means your changes will not be picked up by either skill's reconciliation logic.
- If you need to customize the generated content, remove the marker lines. The subsection becomes yours, but neither skill will update it automatically.
- When you add a new boundary to this file, the first skill you run (either `/program` or `/mock-context`) will emit the Mock Endpoint Specification from your Operations table. You don't need to run both skills in a particular order — demonstration is buildable regardless of which one runs first.
- When you edit a boundary's Operations table, the next skill you run regenerates the subsection. The other skill's artifacts may then be stale; run both skills in any order to bring them into sync.
- When you remove a boundary from this file, run `/mock-context` to remove the Mock Endpoint Specification subsection and the corresponding mock server routes. `/program` does not remove subsections (the Grow loop only creates artifacts as tests need them; removals are a spec operation).

## When to Add Boundaries

Add a boundary when you realize (while writing scenarios or reviewing the domain) that the application needs to interact with something external. Common signals:

- A scenario says the system "sends," "notifies," "charges," "verifies," or "fetches from" something outside itself
- A precondition says "the HR system contains..." or "the payment provider has..."
- You know the application will need to call an API that you don't own

You don't need to list every boundary before starting. Add them as they become apparent. The process handles new boundaries gracefully — if a scenario references an external system that isn't in this file, the process will ask you to add it.

## What NOT to Put Here

- **Internal modules** — This file is for external systems, not internal architecture. The process discovers internal module boundaries through the grow-split-merge loop.
- **Databases** — Persistence is a standing facet (mode of interaction), not an external boundary. The database choice goes in `tech-stack-recommendations.md`.
- **UI frameworks** — These are technology choices, not external systems. They go in `tech-stack-recommendations.md`.
- **API details** — Don't put URL patterns, HTTP methods, or request/response schemas in the user-authored portion of a boundary entry. Those are implementation details. For the real external API, they belong in the production plumbing module. For the mock server, they live in the `### Mock Endpoint Specification` subsection — which is a derived artifact written by `/program` or `/mock-context` (whichever runs first), not by you. The user-authored portion of this file describes boundaries in domain terms.
